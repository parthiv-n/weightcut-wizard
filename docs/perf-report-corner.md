# Corner Tab — Performance Report

Static analysis of the Corner (Community) social tab as shipped. Read-only; no runs, no edits.

## Findings

| Issue | Severity | File:line | Recommended fix |
|---|---|---|---|
| `motion.img` uses framer-motion `layoutId` on every polaroid in the stack — plan explicitly said avoid `layout`/`layoutId` in the deck. `layoutId` enables shared-layout measurement on every render, which is exactly the layout-thrash path the spec wanted to dodge. | **P0** | `src/components/community/PolaroidCard.tsx:103` | Drop `layoutId` from the stack image. Keep `layoutId` only on the profile-grid `PostGrid` tile (the *source* of the shared transition). Add it back to the polaroid lazily, only when a profile nav is imminent. |

Even though `React.memo` is in place, the `animate` prop is a new object literal on each parent render — defeats the memo's intent because motion re-evaluates the animation. | **P1** | `src/components/community/PolaroidCard.tsx:79-83` | Hoist `staticTransform` into a `useMemo` keyed on `(stackPosition, rotationDeg, isTop)`, or compute via static `STACK_OFFSETS` lookup outside the render. |
| Reactivity scope: `useFeedEngagement` mirrors `server.viewerLiked` via 3 separate `useEffect`s that re-fire whenever **any field** on **any post in the page** changes — because the entire `listFeed` page is one reactive payload, a like by another user on post B causes the top engagement hook (bound to post A) to re-sync. ## Question-by-question

1. **DOM cost**: 3 cards max (`STACK_DEPTH = 3`, `posts.slice(topIndex, topIndex+3)`). During a flick there is briefly a 4th node (the exiting card overlay) for 350ms. Memory is bounded — good. Gesture itself doesn't thrash layout because `drag="x"` writes to a motion value bound to `style.x/y/rotate` (composited GPU transform). **But** the `layoutId` on the image and the per-render new `animate` object on background cards cause framer to re-measure on every parent re-render. Plan said "avoid layout props" — `layoutId` is technically *not* the `layout` prop, but it triggers the same shared-element measurement pipeline and *should* be considered a violation.

2. **Image bandwidth**: Full-resolution everywhere on the stack (no thumbUrl in `listFeed` response). Top-3 are preloaded via `new Image()`. Preload IS bounded to top-3 via `visible.slice(topIndex, topIndex+3)`. The Profile `PostGrid` correctly uses `thumbUrl` (line 50, `src/components/community/PostGrid.tsx`); the polaroid stack does not.

3. **Convex reactivity**: One paginated `listFeed` is the reactive boundary. Any like/comment on **any post in that page** invalidates the whole subscription and pushes a new array of 12 rows. Every `PolaroidCard` instance receives a new `post` object identity → `React.memo` saves rerenders via field comparison, but `useFeedEngagement`'s 3 sync-effects all fire and can stomp optimistic state mid-tap.

4. **N+1 in listFeed**: Per page (12 docs default): 1 paginate + 12×(profile query + session get + viewer-like probe + media URL + avatar URL) = **1 + 12×5 = 61 reads**. Author dedupe NOT done — same author on multiple posts pays for repeated profile reads + avatar URL resolves. `listProfilePosts` deduped (line 466 comment); `listFeed` did not.

5. **Optimistic correctness**: Single tap fine. Two taps in the same frame both read the same `liked`/`likeCount` snapshot via closure → double-mutation, server idempotency saves correctness but burns 2 writes. The `doubleTapLike` branch skips the mutation when already liked (good). The `useEffect` re-sync from server (lines 60-62) can roll back an optimistic flip if a server update for an unrelated field on the same page lands before the mutation resolves.

6. **Bundle**: Net-new deps appear to be just `@capacitor/share` (already in `package.json`) and `@capacitor/filesystem` — both dynamic-imported lazily inside the share path (`src/hooks/useMadeWeightShare.ts:69-72`, `src/components/community/MadeWeightShareSheet.tsx:39`). `date-fns` already in tree. **Community is correctly lazy-loaded** (`src/App.tsx:57`) and preloaded 3s after dashboard idle (line 78). Good.

7. **Made Weight Canvas**: Main-thread. `OffscreenCanvas` is *constructed* but never transferred to a worker. `convertToBlob` runs on the same thread that handles UI. Block time ≈ 80-150ms on iPhone 12-class hardware for 1080² + 2 image loads.

8. **Critical path / cold launch**: Order is — Auth resolve → route to `/community` → lazy chunk load (already warm from line 78 preload) → `useMyGyms` resolves → `useGymFeed` opens websocket subscription → `StackSkeleton` renders immediately → first `listFeed` payload arrives (~60 doc reads server-side) → 3 polaroid `<img>` requests fire to Convex Storage CDN. First polaroid pixels gated on the slowest of those 3 storage URLs.

## Single highest-leverage perf win

**Add `thumbUrl` to `listFeed` and stop fetching the 1024px full image for the 2 background cards.** One server change (return `thumbUrl` from `thumbStorageId` like `listProfilePosts` already does) + one client change (use `thumbUrl` for `stackPosition !== 0`). Cuts cold-launch bandwidth by ~70% (3× 200KB → 1× 200KB + 2× 8KB) and lets first polaroid pixels paint sooner because the top card no longer competes with two unrelated full-res fetches. Same code path that the Profile grid already uses — pure parity, very low risk.

## Already-good — don't touch

1. **Stack depth bounded to 3** (`PolaroidStack:39, 105`). Memory + DOM cost stays flat regardless of feed size. The deliberate avoidance of `AnimatePresence` keyed on `topIndex` (lines 21-28) is the right call.
2. **`PolaroidCard` custom equality** (`PolaroidCard.tsx:167-179`) compares only the fields that affect render, omitting unstable refs like `onAuthorLongPress`. Exactly the right shape for a hot-path memo.
3. **Lazy + idle-preload of the Community route** (`App.tsx:57, 78`). Dynamic import dodges the cold-start bundle cost, and the 3s-after-dashboard preload means the chunk is hot by the time a user taps the Corner nav icon.
